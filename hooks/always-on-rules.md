# Always-on safety & discipline rules (7-yod-kuman)

เนื้อหานี้ inject เข้าทุก session อัตโนมัติผ่าน SessionStart hook — ไม่ต้องเรียกด้วย skill หรือ agent

## 1. Simple Code — เขียนให้เรียบง่าย ไม่ซับซ้อน

> เป้าหมาย: โค้ดที่ **อ่านแล้วเข้าใจทันที** และ **แก้น้อยที่สุด** สำหรับงานที่ user ขอ

### หลักการ (บังคับ)

1. **KISS** — เลือกวิธีที่ง่ายที่สุดที่ทำให้ requirement ผ่าน
2. **ขอบเขตแคบ** — แก้/เพิ่มเฉพาะส่วนที่เกี่ยวกับงานนี้ ไม่ refactor รอบข้าง
3. **อ่านง่ายกว่าฉลาด** — ชื่อตัวแปร/ฟังก์ชันบอกความหมาย; ลำดับ logic เป็นเส้นตรง หลีกเลี่ยง clever trick
4. **ทำตาม convention ใน repo** — ดูไฟล์ใกล้เคียงก่อนเขียน อย่าสร้าง pattern ใหม่ถ้าไม่จำเป็น
5. **ไม่เดา requirement** — ถ้าไม่ชัด ถามก่อน implement

### ห้าม (ยกเว้น user ขอชัดเจน)

| ประเภท | ตัวอย่างที่ไม่ควรทำ |
|--------|---------------------|
| Over-engineering | abstraction / factory / wrapper สำหรับโค้ดที่ใช้ครั้งเดียว |
| Helper ฟุ่มเฟือย | ฟังก์ชัน 1–2 บรรทัดที่ inline อ่านง่ายกว่า |
| Feature ล่วงหน้า | flag, config, plugin point ที่ยังไม่มี use case จริง |
| Error handling เกิน | try/catch หรือ fallback สำหรับเคสที่เกิดยากมาก/เป็นไปไม่ได้ |
| Logic ใน JSX | nested ternary, `.map().filter().reduce()` ยาวใน render — ย้ายไป hook/ตัวแปรก่อน return |
| Diff กว้าง | จัด format, rename, "ปรับปรุง" ไฟล์ที่ไม่เกี่ยวกับงาน |

**สรุป:** เขียน code ให้ **เรียบง่าย อ่านง่าย แก้น้อย** — ซับซ้อนเมื่อ requirement หรือขนาดไฟล์บังคับเท่านั้น

## 2. No Bulk Delete of Working Files

เพื่อป้องกันการสูญหายของงานที่กำลังทำอยู่และลดความเสี่ยงจากความเข้าใจผิด:

### บังคับ

1. **ห้ามสั่ง "ลบไฟล์ทั้งหมดที่แก้/สร้างในงานนี้"** หรือทำการลบแบบกว้าง (bulk delete) เช่น:
   - ลบทั้งโฟลเดอร์ของฟีเจอร์ที่เพิ่งทำ
   - ลบไฟล์ทั้งหมดที่เพิ่งแก้
   - ลบทุกไฟล์ที่ `git status` แสดงว่าเปลี่ยนแปลง

2. ถ้าจำเป็นต้องลบ ให้ทำได้เฉพาะกรณีที่ผู้ใช้ **ระบุชัดเจนแบบเจาะจง** ว่า:
   - อยากลบ "หน้า/ฟีเจอร์/ไฟล์" ไหนออก (เช่น path ของไฟล์/โฟลเดอร์)
   - หรือระบุรายการที่ต้องลบเป็น **รายไฟล์**

3. ถ้าผู้ใช้พูดกว้างๆ เช่น "ลบที่แก้ทั้งหมดออก" แต่ **ยังไม่ระบุรายการ** — ให้ **หยุด** และขอให้ผู้ใช้ระบุไฟล์/หน้าที่ต้องการลบก่อนเสมอ

**สรุป:** ห้าม bulk delete ไฟล์ที่กำลังแก้ → ลบได้เฉพาะเมื่อผู้ใช้ระบุชัดเจนว่าให้ลบหน้า/ไฟล์ไหน

## 3. Jira/Issue Card Read Gate

เมื่อ user **ส่งการ์ด / issue / ลิงก์ Jira (หรือ tracker อื่น)** มาให้อ่านเป็นข้อมูลประกอบงาน (requirements, AC, bug context):

### บังคับ

1. **ต้องอ่านเนื้อหาการ์ดให้ได้ก่อน** (ผ่าน MCP, API, หรือข้อความที่ user วางในแชทให้ครบถ้วน) — รวมถึงรูปภาพ/attachment และ comment ทั้งหมด ไม่ใช่แค่ description

2. **ถ้าอ่านไม่ได้** — เช่น เรียก MCP/API ไม่สำเร็จ, ไม่มีสิทธิ์, issue ว่าง, parse ไม่ได้, หรือได้แค่ title/key โดยไม่มีรายละเอียด:
   - **ห้ามแก้ code ใดๆ** ใน repo (รวม refactor เล็กน้อยที่ไม่เกี่ยวกับคำขอ)
   - **ห้ามสร้างไฟล์ logic/UI ใหม่** เพื่อดำเนินตามงานจากการ์ดนั้น
   - ให้ตอบกลับชัดเจนว่าอ่านไม่ได้เพราะอะไร และขอสิ่งที่ต้องการ

3. **เมื่ออ่านได้แล้ว** — ต้อง **สรุปความเข้าใจ** (requirement, AC, edge case ที่เห็นจาก comment/รูป) ให้ user เห็นก่อน และรอ **confirm ชัดเจน** ก่อนเขียนสเปกหรือแก้ code ใดๆ — ห้ามอ่านจบแล้วไปเขียนสเปก/code ต่อทันที จุดไหนในการ์ดที่คลุมเครือหรือขาดรายละเอียด ต้องถาม user ตรงจุดนั้นก่อน ห้ามเดา

4. **เมื่อ user confirm ความเข้าใจแล้ว** — จึงเริ่มวิเคราะห์งาน วางแผน และแก้/เขียน code ตามปกติ

**ข้อยกเว้น:**
- การแก้ไขที่ user สั่งชัดแจ้งแยกจากการ์ด และไม่ขึ้นกับเนื้อหาที่ยังอ่านไม่ได้ — ทำได้ตามคำสั่งนั้น
- **การ์ดประเภท Bug** — ยังต้องอ่านการ์ดให้ครบก่อนเสมอ (ข้อ 1-2 บังคับเหมือนเดิม) แต่ **ไม่ต้องรอ user confirm** ก่อนแก้ code — ดู Bug Fast Path ใน `/spec`

**สรุป:** อ่านการ์ดไม่ได้ → หยุดแก้ code จนกว่าจะมีข้อมูลจากการ์ดครบและอ่านได้
